# DocuWeave

Modern Retrieval-Augmented Generation (RAG) systems are only as good as their document preprocessing pipeline.

Yet most PDF loaders today are fundamentally flawed for RAG use cases.

DocuWeave was built to solve this problem.

It is a layout-aware document parser that converts PDFs into structured, hierarchical, LLM-ready JSON optimized for embedding and retrieval.

---

## The Problem with Existing PDF Loaders

Most PDF parsing tools:

- Extract raw text blobs  
- Ignore layout structure  
- Destroy section boundaries  
- Merge headings and paragraphs  
- Produce poorly segmented chunks  
- Provide no explainability  

For RAG pipelines, this is catastrophic.

When structure is lost:

- Retrieval precision drops  
- Context windows get polluted  
- Chunk embeddings become semantically inconsistent  
- Section-aware filtering becomes impossible  

The majority of RAG failures originate not in the model, but in preprocessing.

DocuWeave addresses this root cause.

---

DocuWeave was built with six principles:

1. Deterministic parsing over AI guessing

3. Hierarchical section reconstruction  
4. Token-aware chunking

No LLMs required.  
No heuristics hidden behind AI calls.

## High-Level Pipeline

DocuWeave follows a clean transformation chain:

```

PDF
→ Layout Blocks
→ Section Hierarchy
→ Token-Aware Chunking
→ Structured JSON Output

````

Each stage is modular and testable.

---

## Step 1: Layout Extraction

We use PyMuPDF to extract:

- Text blocks  
- Bounding boxes  
- Font sizes  
- Page numbers  
- Position metadata  

Instead of flattening content, we preserve layout information.

Each block contains:

- Raw text  
- Font metadata  
- Positional information  
- Page reference  

This becomes the foundation for hierarchy detection.

---

## Step 2: Section Hierarchy Detection

One of the hardest problems in PDF parsing is heading detection.

DocuWeave uses:

- Font size clustering  
- Font weight heuristics  
- Relative position analysis  
- Repetition patterns  

To build a semantic tree:

- Level 1 → Chapter  
- Level 2 → Section  
- Level 3 → Subsection  

The result is a proper hierarchical representation instead of flat text.

---

## Step 3: Token-Aware Smart Chunking

Naive chunking splits by:

- Fixed character count  
- Paragraph boundaries  
- Sliding windows  

DocuWeave instead:

- Preserves section integrity  
- Respects token limits  
- Tracks token counts per chunk  
- Avoids breaking semantic boundaries

### Example usage

```python
from docuweave import parse

doc = parse("sample.pdf")

chunks = doc.to_chunks(max_tokens=500)
````

Each chunk contains:

* Token count
* Section title
* Section level
* Page range
* Unique ID

This enables explainable retrieval.

DocuWeave produces embedding-ready structured JSON.

### Example Output

```json
{
  "metadata": {
    "source": "sample.pdf",
    "total_pages": 120
  },
  "sections": [
    {
      "id": "...",
      "title": "Chapter 1",
      "level": 1,
      "blocks": [],
      "subsections": []
    }
  ],
  "chunks": [
    {
      "id": "...",
      "text": "...",
      "tokens": 487,
      "section_title": "Chapter 1",
      "section_level": 1,
      "page_start": 3,
      "page_end": 5
    }
  ]
}
```

This format supports:

* Metadata filtering
* Section-aware retrieval
* Explainable chunk tracing
* Hybrid search systems

## Why Deterministic > AI Guessing

Many tools attempt AI-based semantic chunking.

This introduces:

* Non-determinism
* Increased cost
* Debug difficulty
* Latency

DocuWeave is deterministic by default.

AI enrichment can be layered on top — not baked into the core parser.

This ensures:

* Predictability
* Testability
* Reproducibility
* Performance
